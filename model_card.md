Model Card: MixPix (Applied AI Edition)

1. Model NameMixPix v2.0: Local Expert Discovery System

2. Intended UseMixPix is designed to provide context-aware music recommendations. Unlike standard recommenders that only look at numbers, this system uses a Knowledge Base to explain the "why" behind a recommendation.Target Audience: Students and music enthusiasts looking for curated "vibes" backed by musicology insights.Context: Built for the Howard University Applied AI course to demonstrate RAG (Retrieval-Augmented Generation) and reliability engineering.
3. How the Model Works
The system uses a hybrid reasoning architecture:The Math Engine: Uses a weighted similarity algorithm to calculate scores based on genre (30%), mood (25%), and audio features like energy and valence.The RAG Engine (New): A keyword-based retriever that identifies user intent (e.g., "coding," "gym") and "retrieves" domain-specific text from a local knowledge base.The Integration: It combines the retrieved text with the mathematical recommendation to generate a final "Expert Insight."
4. DataStructured Data:
 A songs.csv catalog containing 15-16 verified tracks across 12 genres.Unstructured Data (Knowledge Base): Three custom-written musicology files (studyjams.txt, workoutjams.txt, feelthefeels.txt) that provide the "Expert Insights."Data Selection: Songs were selected to provide high-contrast data points for testing energy levels (e.g., Midnight Coding at 0.42 energy vs. Iron Tide at 0.97).
 5. StrengthsGrounding: 
 By using local text files, the system avoids the "generic" feel of standard AI and provides specific, curated advice for Howard students (e.g., focus tips for CS projects).Speed & Privacy: Because it uses a local reasoning engine, it requires no external API keys, has zero latency, and keeps user data entirely private.
 6. Limitations and BiasKeyword Dependency: 
 The "Expert Insight" is only as good as the keyword matcher. If a user uses slang or terms not in the recommender.py logic, the system falls back to general musicology.Small Sample Size: With only 15 songs, the "Reliability Guardrail" is extremely effective, but the system's diversity is low.Binary Matching: As noted in v1.0, the 55% weight on exact-match genre/mood can still lead to "filter bubbles" where the user isn't introduced to adjacent genres.
 7. Reliability & Evaluation
 To ensure the system works as intended, I implemented:The Verification Guardrail: A function that intercepts every recommendation to ensure the song title exists in the songs.csv.Adversarial Testing: I tested the system with "hallucination prompts" (requests for songs not in the library). The guardrail successfully caught 100% of these cases, triggering a safe fallback.Confidence Level: The system maintains high confidence in the "Math Match," but I learned that RAG accuracy is highly dependent on file pathing—requiring an absolute path fix to ensure the knowledge base was always reachable.
 8. AI Collaboration ReflectionHelpful Suggestion: 
 During development, the AI suggested using a BASE_DIR absolute path approach. This was critical because my original relative paths failed when running the script from different folders.AI Flaw/Error: Initially, when asked to generate recommendations, the AI suggested songs that were popular in the real world but not in my dataset. This flaw directly inspired my Reliability Guardrail, teaching me that "creative" AI must always be tethered to a "source of truth" (the CSV).
